配置流程

创建配置记录

通过综合配置页面或直接执行SQL插入 TJ_TJFX_CONFIG 表，创建统计配置记录

INSERT INTO TJ_TJFX_CONFIG (ID, NAME, PROC_NAME, TEMPLATE_NAME, PARAMETERS) VALUES (...)

配置参数映射

定义存储过程参数与前端控件的映射关系，包括：显示名称、默认值、选项配置、是否多选等

配置模板页面（可选）

如果需要自定义展示样式，创建独立的HTML模板页面（如 rad_workload.html）

加载统计菜单

statistics.html 页面初始化时调用 /get-proc-configs API加载所有配置，动态生成左侧菜单

用户选择统计项

用户点击菜单项后，根据配置决定是：加载自定义模板页面 或 直接调用存储过程

执行查询

收集前端参数，调用 /execute-stored-proc API执行存储过程，返回JSON数据

渲染结果

根据返回数据动态渲染表格或调用模板页面的初始化函数

数据流转关系图

TJ_TJFX_CONFIG 表 → StatConfigService → /get-proc-configs API

statistics.html → 动态菜单 → 用户选择 → 加载 rad_workload.html 或直接渲染

/execute-stored-proc → 执行 PROC_NAME → 返回JSON → 前端渲染

配置管理API

API路径 方法 功能描述 参数
/get-proc-configs GET 获取所有启用的统计配置列表 无
/get-proc-config?id=xxx GET 根据ID或名称获取单个配置详情 id: 配置ID或名称
/save-proc-config POST 保存或更新统计配置（MERGE操作） 完整配置JSON对象
/delete-proc-config?id=xxx POST 删除指定的统计配置 id: 配置ID
/save-param-config POST 保存参数配置（仅更新PARAMETERS字段） configId, parameters

数据执行API

API路径 方法 功能描述 参数
/execute-stored-proc POST 执行指定的存储过程并返回结果 procName: 存储过程名
parameters: 参数对象
/get-proc-metadata POST 获取存储过程的参数元信息 procName: 存储过程名
请求示例
{ "procName": "usp_TJFX_RadiologyWorkloadStatistics", "parameters": { "@StartDate": "2026-01-01", "@EndDate": "2026-01-31", "@StatisticsType": "报告医生" } }

核心函数

函数名 功能描述

rad_workload.html - 模板页面规范

必须实现的函数

函数名 功能描述 参数
init(config) 初始化函数，接收配置对象 config: 配置信息（含参数、存储过程名等）
getProcName() 获取主存储过程名（优先读取mainProcName） 无
fillParamsFromConfig(paramsJson) 从配置填充参数到页面控件（大小写不敏感） paramsJson: 参数JSON字符串
loadData() 加载数据（不应在页面加载时自动调用） 无
参数传递规范
  • 参数名统一使用大写开头（如 @StartDate）
  • 参数匹配采用大小写不敏感方式
  • 配置中的参数名应与存储过程参数名完全一致
  • 日期类型参数统一使用 YYYY-MM-DD 格式

事件触发条件

事件名称 触发条件 处理逻辑

参数传递流程

配置层参数定义

TJ_TJFX_CONFIG.PARAMETERS 字段存储JSON格式的参数配置

[ {"name":"@StartDate","displayName":"开始日期","type":"datetime","defaultValue":"2026-01-01","isRequired":true}, {"name":"@EndDate","displayName":"结束日期","type":"datetime","defaultValue":"2026-01-31","isRequired":true}, {"name":"@StatisticsType","displayName":"统计类型","type":"varchar","options":"报告医生,审核医生,技师","defaultValue":"报告医生"} ]

前端动态渲染

renderCustomParamInputs() 解析JSON，根据类型生成日期选择器、下拉框或文本框

参数收集

查询时遍历 .param-value-input 控件，构建参数对象

API调用

POST到 /execute-stored-proc，包含 procName 和 parameters

数据库执行

服务端构建SQL命令，执行存储过程，返回DataTable并序列化为JSON

存储过程调用逻辑

主存储过程调用

当配置中存在 TEMPLATE_NAME 且不为空时：

  1. 加载独立HTML模板页面
  2. 调用模板的 init(config) 函数传入配置
  3. 模板内部调用 getProcName() 获取主存储过程名
  4. 模板内部调用 loadData() 执行查询

明细存储过程调用

仅当 HAS_DETAIL_PROC = 1 时才调用明细存储过程：

  1. 用户点击汇总数据行触发
  2. 调用 DETAIL_PROC_NAME 指定的存储过程
  3. 传入额外的筛选参数（如科室、医生等）
存储过程命名规范
  • 主存储过程：usp_TJFX_XXXStatistics 或 proc_XXX
  • 明细存储过程：usp_TJFX_XXXStatistics_Detail 或 proc_XXX_Detail
  • 配置中 PROC_NAME 应为主存储过程名
  • 模板页面通过 mainProcName 字段获取主存储过程名

2026-05-06 问题记录与修改错误示范

核心问题：参数值重复导致默认值覆盖

问题现象：用户修改参数后，发送给后端的仍然是配置的默认值，而不是用户修改后的值。

错误示范一：硬编码存储过程名称

错误代码：

function getProcName() { return 'sp_RadiologyWorkloadStatistics'; // 硬编码！ }

问题原因：破坏了模块化设计原则，存储过程名称应该从配置中获取。

正确做法：

function getProcName() { return currentConfig?.mainProcName || currentConfig?.procName || 'proc_RadiologyWorkload'; }
错误示范二：硬编码默认值覆盖配置

错误代码：

<input type="date" id="startDate" value="2026-01-01"> <!-- HTML中硬编码 -->

问题原因：HTML中硬编码的默认值会覆盖配置中的默认值，且无法动态更新。

正确做法：

<input type="date" id="startDate"> <!-- 不设置value，由JS从配置填充 -->
错误示范三：重复生成参数输入框

错误代码：

// statistics.html 中的错误逻辑 if (config.parameters && (!config.templateName || !config.templateName.trim())) { renderCustomParamInputs(config.parameters); // 生成输入框 } if (config.templateName && config.templateName.trim()) { loadCustomTemplate(config.templateName, [], config); // 加载模板 // 模板中又有自己的输入框，导致重复！ }

问题原因：当有自定义模板时，statistics.html 仍然会生成参数输入框，导致页面上有两个相同ID的输入框。

正确做法：

// 修改后的逻辑 - 互斥处理 if (config.templateName && config.templateName.trim()) { // 有自定义模板时，直接加载模板，由模板负责参数管理 loadCustomTemplate(config.templateName, [], config); } else if (config.parameters) { // 没有自定义模板时，才生成参数输入框 renderCustomParamInputs(config.parameters); }
错误示范四：jQuery选择器匹配多个元素

错误代码：

// 由于有两个相同ID的输入框 $('#startDate').val() // jQuery获取到的是第一个，但用户看到的是第二个

问题原因：页面上有两个相同ID的输入框，jQuery获取到的是DOM中的第一个，而不是用户看到和修改的那个。

调试方法：

console.log('匹配到的 startDate 元素数量:', $('input[id="startDate"]').length); // 应该显示 1，如果显示 2 则说明有重复
错误示范五：$(document).ready 无条件覆盖默认值

错误代码：

$(document).ready(function() { var today = new Date().toISOString().split('T')[0]; $('#startDate').val(today); // 无条件覆盖配置默认值 });

问题原因：$(document).ready 在页面加载时会无条件设置日期为今天，覆盖了配置的默认值。

正确做法：

$(document).ready(function() { var today = new Date().toISOString().split('T')[0]; if (!$('#startDate').val()) { // 只在为空时设置 $('#startDate').val(today); } });

正确的参数传递流程

  1. 页面加载：从配置中获取默认值并设置到输入框（仅当输入框为空时）
  2. 用户修改：用户可以修改输入框的值
  3. 执行查询：获取输入框的当前值（而不是配置的默认值）
  4. 传递给后端：将当前值作为参数传递

设计原则总结

  • 配置驱动：存储过程名称和默认值应从数据库配置读取
  • 模块化：自定义模板应完全负责自己的参数管理，statistics.html 不应生成冲突的输入框
  • 避免硬编码：所有可配置的值都不应硬编码在代码中
  • 唯一性：同一页面中相同ID的元素只能有一个
  • 参数优先：用户修改的值优先于配置的默认值
